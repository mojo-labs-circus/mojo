Anatomy of a Mojo system

A Mojo system is a personal AI. A Jarvis: always on, remembers you, reachable from every device you own, and yours. It is assembled from swappable pieces the way a Linux system is: distributions, desktops, and shells compete and get replaced, while your home directory stays yours. The Mojo System Interface (MSI) is the standard that defines each piece and each seam between pieces, so that anyone can build a better implementation of any piece and you can run whichever you want.

One rule underneath everything: every box below is a replaceable implementation, except one. Your data. The data is the heart and soul of the system: package it up, drop it into a different assembly of pieces, and it is still the same assistant, the way dotfiles make any machine yours.

Defined by the MSI: the standard's own work Adopted: an existing standard, consumed as-is Deliberately not standardized: implementations compete here
A human

The Owner

One person. Inside the system they are represented by a verifiable credential, something key-pair or DID-shaped, so any piece can check that “the owner said so” actually means the owner said so. Every permission in the system bottoms out here.

aOwner verification: how any surface proves the human behind it is the owner
bProactive delivery: how the system reaches the owner unprompted
Swappable piece, profile: Client

Windows

Every surface the owner talks through. Each is a window into the same one assistant, never a separate copy of it. A window can run on one of the owner’s machines (a terminal, a desktop app) or on a thin device that runs nothing else (a phone app, a watch). Windows are implementations too: anyone can build one, and an owner can use any mix at once.

terminal, on a machinephone, thinvoicewatch, thin…any window
Swappable piece, profile: Peripheral

Peripherals

Every capability the system senses or acts through on the owner’s devices: camera, location, contacts, notifications, screen control. A phone is usually both a window and a peripheral host, but talking through a device and commanding it are two different jobs with two different contracts. A peripheral declares what it offers; every invocation of one is a consequential action and goes through the kernel.

cameralocationcontactsnotificationsscreen control…any capability
cClient contract: what a window is handed, what it may cache, what it must never hold on its own
oPeripheral contract: what a capability declares, how it is invoked, every invocation kernel-checked
One of the owner's machines: laptop · desktop · server
Runs: ephemeral. One agent runtime working one task, killable, several live at once
Swappable piece, profile: Harness

Agent runtime

The loop that does the thinking and acting: an existing agent runtime, adopted whole. The one piece that is born and dies with the run itself. The MSI defines only how it connects, what it is handed and what it owes back. Its insides are its own business.

The runtime’s internals (how it plans, manages context, retries) are deliberately not standardized. Runtimes compete there; the MSI only pins the connections. It reads Skills (SKILL.md) straight from the identity the same way it reads any other data, not as a per-run binding like the services below — skills aren’t chosen per run, they’re just there.
hSkill format: SKILL.md
eModel wire contract: the OpenAI-compatible baseline all local servers already speak, plus declared optional capabilities
fTool protocol: MCP
sMemory access: what a run asks of a provider and what every compliant provider must accept and return
qSandbox contract: where an allowed action executes and what it can see while running
Services: plural. Several of each kind can serve one machine at once; which ones a run uses is the router’s pick
Swappable piece, profile: Model endpoint

Models

Local or self-hosted model servers such as vLLM, Ollama, or llama.cpp. Any endpoint speaking the standard wire shape is interchangeable. Endpoints are modality-typed (chat, embedding, image, speech), separately swappable, and a runtime may use different endpoints for different steps of the same run.

Consumed format

Tools

MCP servers. What a tool may actually touch is decided by the kernel, not by the tool protocol.

Swappable piece, profile: Memory Provider

Memory providers

The software that serves the data: retrieval, indexing, search, the views a runtime works from. Because the data stays in the standard shape, several providers can serve the same identity at once, a vector-heavy one beside a grep-style one, and a run is pointed at whichever suits its task. Providers derive, they never own: anything built from the data must be rebuildable from the data alone. The relationship is Obsidian and your vault. Delete the app, keep everything. What a run asks of a provider and what it gets back is its own seam (s); whatever path a write takes, it lands in the data’s shape, which is m’s business.

Swappable piece, profile: Sandbox

Sandboxes

Where a permitted action actually executes: a container, an isolated process, another of the owner’s machines, a remote service. Placement, not permission. Which actions are allowed at all is the kernel’s decision; where an allowed action runs, and what it can see while running, is the sandbox’s.

System control: singular. Exactly one live instance of each per machine, swapped by the owner, never per run
Swappable piece, profile: Router

Router

Always on. When nothing is running, the router is what watches the triggers: the clock, outside events, idle time. When one fires, or the owner asks for something, it assembles the run: which agent runtime, which model, which memory provider, which sandbox, on which machine, within the owner’s constraints and budget. Every binding is chosen per run, and sub-runs get their own selection, recursively. Routers compete on selection quality. Hard limits are the owner’s policy data, which a router obeys. They are never properties of the router itself.

dRouting contract: task + roster + constraints in, selection out
gTriggered runs: how a run fires from the clock, an outside event, or idle time; downtime memory maintenance is just a run; triggers themselves are data
Swappable piece, profile: Kernel

Kernel

enforcement

The enforcement layer. Every consequential action (reading the data, writing it back, touching a tool or a peripheral that reaches the outside world) passes through here and is checked against what the owner allowed. It grants, narrows, and revokes permissions, and records what happened. Each machine runs its own kernel instance so enforcement keeps working offline, but all of an owner’s machines answer to one shared policy. Like every other piece, the kernel is swappable: the MSI pins the contract precisely so that better kernels can replace any given one.

iRun contract: what a runtime is handed (identity fragment, capabilities, task), what it owes back (its learnings, written into the data’s shape), and how a live run is interrupted or killed
jEnforcement contract: the permission model, granting, narrowing, revoking, auditing
Swappable piece, profile: Credential Broker

Credential broker

Custody of the owner’s secrets: API keys, tokens, passwords. It stores them, scopes them, and injects them into the specific call that needs them, so a real secret never sits in a model’s context or a runtime’s hands. The kernel decides whether an action is allowed; the broker makes sure the secret reaches only that action. Custody is exactly the kind of fact two live instances must never disagree about.

Swappable piece, profile: Provenance

Provenance

Trust-tags content before it reaches a run: where it came from, whether the owner wrote it, whether it arrived from the open web. A runtime and a kernel can then treat a stranger’s words differently from the owner’s. One live instance per machine: if a run could pick its own tagger, untrusted content could be laundered through the laxest one.

Swappable piece, profile: Fleet Manager

Fleet manager

Keeps the owner’s machines one system. It admits a new machine into the set, revokes one no longer trusted, and coordinates the machines so one identity stays coherent across all of them: sync, conflict, partition, who’s driving a live task. The kernel enforces on one machine; the fleet manager makes N machines answer as one assistant.

pSecret custody: secrets stored, scoped, and injected into exactly the call that needs them, never into model context
rContent trust: how external content is trust-tagged before it reaches the model
mData shape: several providers can serve the same data at once; providers derive, never own. Writes land in the data’s standard shape and reconcile there
Not a piece: the data. The only thing never swapped

The identity

portable · yours

Everything that makes the assistant this assistant: its memory of you, its personas, its skills (SKILL.md, learned procedures a runtime reads straight off the data), the owner’s policies and budgets, a record of who wrote what under what authority. The memory is one; the personas on top of it can be many — each a voice plus a scoped view over the same memory, cut by per-unit permissions the data itself carries, so a work persona and a casual persona are two characters reading from one brain, not two brains. Permissions travel with the data itself. Stored as plain files. It can be packaged up and dropped into an entirely different assembly of pieces and it is still the same assistant. This is what makes any Mojo system yours.

Built on a small set of file-like primitives the MSI defines: the equivalent of files and directories in an OS.
HOST OS: any Linux distribution (macOS later). Every piece above, the kernel included, is ordinary software running in OS userspace. No kernel modules, no custom OS.
kFederation: machine ↔ machine, one identity kept coherent across all of them (sync, conflicts, and who’s driving a live task)
nMachine admission: how a new machine earns its way into the owner’s set
lAgent-to-agent: A2A, for talking to agents outside the owner’s system

The owner’s other machines

Same identity, same policy. A desktop, a server, a laptop: together they behave as one assistant, not several that happen to share notes.

Other people’s agents

Different owners, opaque by design. A2A carries the conversation; nothing about the protocol extends trust. This is also how one person’s Jarvis speaks to another’s.

What survives, what swaps

The whole design reduces to one split.

The identity is owed literal equivalence

Same memory, same permissions, same relationship, bit for bit, across any swap of any piece. This is the only equivalence the system guarantees anywhere.

Everything else is owed nothing

A different model thinks differently. A different runtime plans differently. A different window looks different. No piece is owed identical output; only the identity is owed identical data. That freedom is why pieces can compete at all.

Seam ledger

Every arrow in the diagram and what the MSI must pin down for it. Eleven pieces get conformance profiles (Client, Peripheral, Router, Harness, Model endpoint, Sandbox, Credential Broker, Provenance, Kernel, Memory Provider, Fleet Manager), meaning an implementation is compliant against its profile, never against the whole document. The identity is data, not a piece: nothing to conform, only a shape to keep.
