## 1. File Structure — Global and Project

### What's Real

**Global `~/.claude/` — what Claude Code recognises:**

| File / Dir | Auto-loads? | Notes |
|---|---|---|
| `CLAUDE.md` | Yes, every session | Loaded at startup |
| `settings.json` | Yes | Permissions, hooks, env, model defaults |
| `keybindings.json` | Yes | Hot-loaded — changes take effect immediately |
| `rules/` | Conditional | Files with `paths:` frontmatter load only when matching files are opened |
| `agents/` | On demand | User-level subagent definitions |
| `skills/` | On demand | User-level slash commands / reusable prompts |
| `commands/` | On demand | Single-file slash commands (older format) |
| `workflows/` | On demand | Dynamic workflow scripts |
| `output-styles/` | On demand | Custom response style presets |
| `agent-memory/` | On demand | Persistent memory for user-scoped subagents |
| `projects/<project>/memory/` | Yes (first 200 lines / 25KB) | Claude writes its own notes here; auto-loaded at session start |
| `themes/` | Yes | Custom color themes |
| `.claude.json` | Yes | App state, OAuth tokens, personal MCP servers, UI toggles |

**Files that are NOT native Claude Code concepts:**

- **SOUL.md** — Not a Claude Code feature. Community naming convention borrowed from other AI tools, has no special loading behaviour.
- **RULES.md** — Not native. Use `.claude/rules/` directory instead.
- **AGENTS.md** — Not native. Claude Code reads `CLAUDE.md`. Karpathy uses `AGENTS.md` for multi-tool portability across Codex, Cursor, etc. If your repo has one for other agents, create a `CLAUDE.md` that imports it: `@AGENTS.md`. Claude Code docs explicitly address this case.
- **MEMORY.md at global level** — Does not exist. Memory is project-scoped at `~/.claude/projects/<project>/memory/MEMORY.md`, auto-generated by Claude, not a file you create manually in `~/.claude/`.

**Project `.claude/` inside a repo:**

| File | Committed? | Behaviour |
|---|---|---|
| `CLAUDE.md` | Yes | Loaded for every session in this project |
| `settings.json` | Yes | Team-shared hooks, permissions, MCP, model defaults |
| `settings.local.json` | No (auto-gitignored) | Your personal overrides — highest-precedence user-editable scope |
| `.mcp.json` (at repo root) | Yes | Shared MCP servers for the team |
| `.worktreeinclude` | Yes | Gitignore-syntax patterns of files to copy into each worktree |
| `rules/` | Yes | Path-scoped instruction files |
| `skills/` | Yes | Project-level slash commands |
| `agents/` | Yes | Subagent definitions |
| `output-styles/` | Yes | Custom response styles |
| `workflows/` | Yes | Dynamic workflow scripts |
| `agent-memory/` | Yes | Persistent memory for project-scoped subagents |

**CLAUDE.md hierarchy — exact behaviour:**

Claude Code walks *up* from your CWD to the filesystem root, collecting every `CLAUDE.md` and `CLAUDE.local.md` it finds. Files are then injected in root-to-CWD order (most general first). Within each directory: `CLAUDE.md` loads before `CLAUDE.local.md`.

Example: running `claude` from `foo/bar/`:

1. `/CLAUDE.md` (if exists)
2. `foo/CLAUDE.md`, then `foo/CLAUDE.local.md`
3. `foo/bar/CLAUDE.md`, then `foo/bar/CLAUDE.local.md`
4. Any `foo/bar/baz/CLAUDE.md` loads *on demand* when Claude reads a file in `baz/`

After context compaction: **project-root `CLAUDE.md` is re-read from disk and re-injected**. Nested `CLAUDE.md` files are **not** — they reload the next time Claude opens a file in that subdirectory.

Size limits: no documented hard line count. The 200-line guidance is adherence-driven: Chroma's 2025 benchmark showed frontier models drop from 95% to 60% instruction accuracy as context bloats. The system prompt already consumes ~50 of the ~150–200 reliable instructions a frontier model will follow per session.

**`@reference` system:**

- Syntax: `@path/to/file` (relative to the file containing it, or `@~/file` for absolute).
- Imported files expand at session start alongside the CLAUDE.md that references them — not lazy-loaded.
- Recursive imports supported; max depth: 4 hops.
- First use in a project prompts an approval dialog. Declining disables imports permanently for that project (dialog does not reappear).

### What's Convention

- Top-level `SOUL.md`, `AGENTS.md`, `RULES.md` naming borrowed from other tools. Importing them into CLAUDE.md via `@` is the correct Claude Code pattern.
- `agent_docs/` directory for extended specs (seen in community repos, not a native concept).

### Best Practice

- Keep root CLAUDE.md under 200 lines. Move extended content to `@referenced` files or `.claude/rules/` path-scoped files.
- Commit `settings.json` and `.mcp.json`. Gitignore `settings.local.json` and `CLAUDE.local.md`.
- Monorepo strategy: lean root CLAUDE.md covering the whole tree, focused per-package CLAUDE.md files for domain-specific conventions. Use `claudeMdExcludes` in `settings.local.json` to skip ancestor CLAUDE.md files that don't apply when working in a specific package.
- `@reference` large specs and docs rather than inlining them in CLAUDE.md.

### Relevant to Mojo

- `mojo-labs/CLAUDE.md` (existing) covers the framework level. Each component repo (`ringmaster/`, `performer/`, etc.) should have its own `.claude/CLAUDE.md` covering its domain conventions.
- When doing component work from inside that component's directory, Claude picks up both the root and the component-specific file — no manual coordination needed.
- Auto-memory is project-scoped, so `mojo-labs/` will have its own `~/.claude/projects/mojo-labs/memory/` separate from per-component memory if those are separate repos. Worth understanding this boundary as team work begins.
- `claudeMdExcludes` will be useful once `mojos/` and `performer/` have diverging conventions that shouldn't bleed into each other.

## 2. Ways of Working

### Boris Cherny (Creator of Claude Code)

Boris built Claude Code as a side project at Anthropic in September 2024. By December 2025 he was shipping 259 PRs/month. As of mid-2026, Claude Code is 100% written by Claude Code. Sources: WorkOS interview, InfoQ, Engineered Intelligence, Platformer.

**Daily workflow:**

- 5 local Claude Code terminals (numbered 1–5), each on a separate task/worktree.
- 5–10 additional sessions on claude.ai/code in parallel.
- Starts sessions on iOS in the morning, resumes on desktop.
- Sessions moved between local and web via `&` command or `--teleport` flag.
- Outputs: 20–30 PRs per day. ~10–20% of parallel sessions abandoned (unexpected scenarios).
- His framing: *"It's not about deep work. It's about how good I am at context switching and jumping across multiple different contexts very quickly."*

**CLAUDE.md:**

- Boris's personal file: ~2 lines, pointing to the team shared file.
- Team file: ~2,500 tokens. Contains: style conventions, design guidelines, PR template, preferred package manager (`bun` not `npm`), exact typecheck/test/lint commands, error rules accumulated from mistakes.
- Committed to git. Team contributes multiple times per week.
- Quote: *"Anytime we see Claude do something incorrectly, we add it to the CLAUDE.md so Claude knows not to do it next time."*
- Pruning policy: *"Keep it short. If it gets too long: delete it and start fresh."*
- *"With each new model, you need less in the CLAUDE.md."*

**Errors → CLAUDE.md updates:**
Every mistake becomes a rule. Not a conversational correction — a persistent rule. He uses `@.claude` in PR review comments to prompt Claude to add a rule to CLAUDE.md as part of the PR diff. He calls this "Compounding Engineering" — institutional memory built incrementally.

**Plan mode:**
With Opus 4.5: ~80% of sessions started in plan mode (Shift+Tab twice). Iterate until plan is good, then auto-accept edits.
With Opus 4.6+: reportedly shifted away from explicit plan mode. Quote from howborisusesclaudecode.com: *"I don't use that anymore. I use auto mode."* Newer models plan implicitly, making explicit plan-mode overhead rather than necessity.

*Source conflict: earlier sources (early 2026, Opus 4.5 era) describe heavy plan mode use; later sources describe moving to auto. Both are accurate for their period.*

**Verification:**
His #1 tip: *"Give Claude a way to verify its work. If Claude has that feedback loop, it will 2-3x the quality of final results."*

**Model preference:** Opus with extended thinking for everything. *"Despite increased size and processing time, it requires less steering and excels at tool usage."*

**MCP servers used:** Slack, BigQuery, Sentry — giving Claude access to the operational environment.

*Source quality note: Boris's actual CLAUDE.md and subagent files are not publicly posted. All implementation details come from secondhand interview summaries (InfoQ, Engineered Intelligence) and community reconstructions (howborisusesclaudecode.com, the bcherny-claude GitHub repo). The concepts are well-attested; the implementation specifics are inferred.*

### Andrej Karpathy

**Four principles** (packaged as a SKILL.md, went viral — 3,741 stars in one week, users reporting 65% → 94% accuracy improvement):

1. **Think Before Coding** — Don't pick an interpretation and run silently. Surface assumptions explicitly. Present multiple interpretations of ambiguous requests. Recommend simpler approaches. Halt and articulate confusion rather than pushing through.

2. **Simplicity First** — Minimum code that solves the stated problem. Nothing speculative. Prohibited: features beyond stated requirements, abstractions for single-use code, unrequested flexibility, error handling for impossible scenarios. Self-check: "Would a senior engineer consider this overcomplicated?"

3. **Surgical Changes** — Touch only what you must. Don't improve unrelated code, comments, or formatting. Don't refactor functioning code. Maintain existing style even if suboptimal. Flag pre-existing dead code without removing it. Remove resources only if they were made unused by *your specific change*.

4. **Goal-Driven Execution** — Define measurable success criteria before starting. Transform requests into verifiable outcomes. Strong criteria enable independent iteration; weak ones ("make it work") require ongoing clarification. Quote: *"LLMs are exceptionally good at looping until they meet specific goals. Don't tell it what to do, give it success criteria and watch it go."*

**AGENTS.md:** Karpathy uses this filename for multi-tool portability (Codex, Cursor, Claude Code). Not a Claude Code native file. Import with `@AGENTS.md` in CLAUDE.md, or symlink.

**LLM Wiki pattern** (April 2026 gist, 5,000+ stars):

Three-layer architecture:

- `raw/` — immutable source documents (PDFs, articles, transcripts, bookmarks). LLM reads only, never modifies.
- `wiki/` — LLM-generated and LLM-owned markdown: summaries, entity pages, concept pages, cross-references, contradiction flags, `overview.md`, `index.md`, `log.md`.
- Schema document (CLAUDE.md or AGENTS.md) — defines wiki structure, naming conventions, and the three operations.

Three operations:

- **Ingest**: Add source to `raw/`, prompt Claude to process. LLM reads source, creates summary pages, updates `index.md`, revises entity pages. Each ingest touches ~10–15 wiki pages.
- **Query**: Ask questions. LLM searches relevant pages, synthesises answers with citations. High-value query answers become new wiki pages.
- **Lint**: Periodic health check. Identifies contradictions, stale claims, orphaned pages, missing cross-references, data gaps.

Navigation files:

- `index.md`: catalog of every wiki page with one-line summaries, organised by category.
- `log.md`: append-only record with consistent prefixes (e.g., `## [2026-04-02] ingest | Title`) — parseable by Unix tools.

Works well to ~100–200 sources without vector databases. Obsidian as frontend; git for versioning.

*Note: Karpathy's gist is intentionally abstract. The `raw/`/`wiki/` directory names and `index.md`/`log.md` navigation files are well-attested across multiple independent implementations. The exact CLAUDE.md schema is not reproduced verbatim in any public source — the operational details above are community synthesis from the gist.*

### Context Window Management

**What's real:**

- `/compact [hint]` — summarises conversation, replaces history with summary. Lossy. Accepts steering hints: `/compact focus on auth refactor, drop test debugging`. Claude Code inserts a checkpoint before compacting.
- `/clear` — wipes session entirely. You control what carries forward.
- `/rewind` — keeps earlier file reads, drops a failed attempt (useful when direction was wrong but initial exploration was valid).
- Project-root CLAUDE.md **survives compaction** — re-read from disk and re-injected.
- Nested CLAUDE.md files **do not** auto-reinject after compaction.
- Auto-memory (MEMORY.md in `~/.claude/projects/`) always re-injected at session start (first 200 lines / 25KB).
- Skill descriptions are **not** re-injected after compaction — only invoked skills are preserved.

**Practitioner patterns:**

- Compact at **60% context**, not 90%. Quality degrades well before the hard limit (NoLiMa benchmark: 11/12 models below 50% performance at 32k tokens).
- Compact after completed subtasks, not when forced.
- One practitioner (albertsikkema.com) experiments with `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=70` and `CLAUDE_CODE_DISABLE_1M_CONTEXT=1` (caps at 200k). This is a personal experiment, not official guidance.

**Handoff brief pattern (blakecrosley.com — 50-session analysis):**
Before closing a long session, ask Claude to write:

```
Status: [% complete, blockers]
Files changed: [list]
Decisions: [key architectural choices made]
Blocked: [what prevents next steps]
Next: [exact files/tasks for next session]
```

Save to `~/.claude/handoffs/` or CLAUDE.md. One practitioner maintains 49 of these, under 500 tokens each.

### Parallelisation

**What's real — git worktrees:**

- `claude --worktree feature-name` creates a worktree at `.claude/worktrees/feature-name/` on a new branch `worktree-feature-name`.
- Add `.claude/worktrees/` to `.gitignore`.
- `.worktreeinclude` file at repo root: `.gitignore`-syntax patterns of files to copy into worktrees (e.g., `.env` files).
- `worktree.baseRef: "head"` in settings to branch from local HEAD rather than `origin/HEAD`.
- `claude --worktree "#1234"` creates a worktree from PR 1234.
- Subagent frontmatter `isolation: worktree` auto-creates worktrees for parallel subagents.
- Cleanup: no uncommitted changes → automatic cleanup. Otherwise Claude prompts.

**Coordination:**

- Each worktree has a distinct branch. A branch can only be checked out in one worktree simultaneously.
- Each task should touch different files when possible — conflicts at merge time are normal git.
- Avoid `git stash` when using worktrees; use branches instead.

**Boris's decomposition approach:** 5 tasks, 5 sessions, fast context switching. Not deep single-threaded focus.

---

### Long-Running Projects

**Preventing context rot:**

- CLAUDE.md as ground truth: prune actively, don't only append. When a rule becomes redundant (model infers it), remove it.
- Session discipline: compact/clear at task boundaries, not when context forces it.
- Subagent delegation: verbose ops (log analysis, test runs) to subagents that return summaries, keeping main context lean.
- Auto-memory: Claude writes its own learnings. Survives sessions. Trust but verify over time.

**What actually degrades:** Foundational instructions get compressed out as conversations grow. CLAUDE.md re-injection at compaction partially compensates, but conversation-only instructions (things you told Claude mid-session but didn't write down) are gone.

**Rule:** Anything you want Claude to remember across sessions must be in a file, not just said in conversation.

## 3. Hooks — Deep Dive

### What's Real

**All hook events (complete list):**

| Category | Events |
|---|---|
| Session | `SessionStart`, `SessionEnd` |
| Per-turn | `UserPromptSubmit`, `Stop`, `StopFailure` |
| Per-tool | `PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `PermissionRequest`, `PermissionDenied` |
| State changes | `ConfigChange`, `CwdChanged`, `FileChanged`, `WorktreeCreate`, `WorktreeRemove` |
| Context | `InstructionsLoaded`, `PreCompact`, `PostCompact` |
| Other | `Setup`, `UserPromptExpansion`, `MessageDisplay`, `SubagentStart`, `SubagentStop`, `TaskCreated`, `TaskCompleted`, `TeammateIdle`, `Notification`, `Elicitation`, `ElicitationResult` |

**Exit code semantics:**

| Code | Meaning |
|---|---|
| `0` | No objection. Stdout text added to context for some events. Normal permission flow applies. |
| `2` | Blocked. Stderr → Claude as feedback. For some events, execution continues with user notification. |
| Other | Non-blocking error. Execution continues. |

**What each event can do:**

| Event | Can block? | Notes |
|---|---|---|
| `PreToolUse` | Yes (exit 2) | Tool blocked, stderr → Claude. Fires BEFORE permission-mode checks — blocks even in `bypassPermissions` mode. |
| `PostToolUse` | No | Tool already ran. |
| `UserPromptSubmit` | Yes (exit 2) | Turn ends, reason shown. |
| `Stop` | Yes, up to 8x | Exit 2 → Claude keeps working. After 8 consecutive blocks, override cap prevents infinite loop. |
| `PermissionRequest` | Yes (exit 2) | Dialog not shown; reason sent to Claude. |
| `SessionStart` | No (exit 2 shows to user, execution continues) | — |

**Timeouts:**

- Command/HTTP/MCP hooks: 10 minutes (except `UserPromptSubmit`: 30s, `MessageDisplay`: 10s).
- Prompt hooks: 30s. Agent hooks: 60s.
- Override per-hook with `timeout` field.

**Hook input fields (every event):**

```json
{
  "session_id": "abc123",
  "cwd": "/path/to/project",
  "hook_event_name": "PreToolUse",
  "permission_mode": "default",
  "transcript_path": "/path/to/transcript.jsonl"
}
```

Plus event-specific fields (`tool_name`, `tool_input` for tool events; `stop_hook_active` bool for Stop; etc.)

**JSON output format** (for hooks that can produce structured decisions):

```json
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",
    "permissionDecisionReason": "Use rg instead of grep"
  }
}
```

### Patterns

**The core insight (community-documented):** CLAUDE.md instructions have ~80% adherence. Hooks are 100% deterministic. If a rule keeps getting violated, convert it to a hook.

**Auto-format on write (most common hook in the wild):**

```json
{
  "PostToolUse": [{
    "matcher": "Write|Edit",
    "hooks": [{"type": "command", "command": "bun run format || true"}]
  }]
}
```

Boris's exact version uses `bun run format || true`. Variants use `prettier`, language-specific formatters. The `|| true` prevents the hook from blocking on non-critical format failures.

**Stop hook quality gate (run tests before session ends):**

```json
{
  "Stop": [{
    "hooks": [{
      "type": "command",
      "command": "bash -c 'INPUT=$(cat); if [ \"$(echo \"$INPUT\" | jq -r .stop_hook_active)\" = \"true\" ]; then exit 0; fi; npm test --silent || exit 2'",
      "timeout": 60
    }]
  }]
}
```

The `stop_hook_active` check is essential — prevents infinite loops when the hook keeps failing.

**Desktop notifications (macOS) — for parallel session management:**

```json
{
  "Notification": [{
    "hooks": [{
      "type": "command",
      "command": "osascript -e 'display notification \"Claude Code needs attention\" with title \"Claude Code\" sound name \"Ping\"'"
    }]
  }]
}
```

Boris uses system notifications to track when any of his 5+ sessions needs input.

**Block dangerous commands (PreToolUse):**

```json
{
  "PreToolUse": [{
    "matcher": "Bash",
    "hooks": [{
      "type": "command",
      "command": "bash -c 'CMD=$(jq -r .tool_input.command <<< \"$(cat)\"); echo \"$CMD\" | grep -qiE \"(rm -rf /|DROP TABLE|rm -rf ~)\" && { echo \"Blocked: dangerous command\" >&2; exit 2; }; exit 0'"
    }]
  }]
}
```

**PostToolUse audit log:**

```json
{
  "PostToolUse": [{
    "hooks": [{
      "type": "command",
      "command": "jq '{ts: now, session: .session_id, tool: .tool_name, input: .tool_input}' >> /tmp/claude-audit.jsonl"
    }]
  }]
}
```

**Environment injection on directory change:**

```json
{
  "CwdChanged": [{
    "hooks": [{
      "type": "command",
      "command": "direnv export bash > \"$CLAUDE_ENV_FILE\" 2>/dev/null; exit 0"
    }]
  }]
}
```

### Shared Hook Configuration

- Team hooks: `.claude/settings.json` (committed). Every team member gets them.
- Personal overrides: `.claude/settings.local.json` (auto-gitignored). Personal hooks that shouldn't apply to teammates.
- Hook config changes take effect **immediately** — no restart required.

### Security Implications

Hooks run as the user — full filesystem access, full shell. They fire for ALL sessions using that config, including subagents. A compromised hook is effectively arbitrary code execution.

PreToolUse hooks are pre-permission — they bypass `bypassPermissions` mode. This makes them more powerful but also means a misconfigured hook can silently block operations.

**Relevant to a team setup:** Hooks committed to `.claude/settings.json` apply to everyone who clones the repo. Review hook changes in PRs the same way you'd review CI configuration changes.

### Best Practice

- Start with notifications and auto-format (low risk, high value).
- Add Stop hook quality gate once you have a stable test suite.
- Add PreToolUse blocks for specific dangerous patterns as they're discovered.
- Log audit trail for multi-user/AI setups from day one — you want history when something goes wrong.

## 4. MCP Servers

### What's Real

**`.mcp.json` schema (at repo root, committed):**

```json
{
  "mcpServers": {
    "github": {
      "type": "http",
      "url": "https://api.githubcopilot.com/mcp/",
      "headers": { "Authorization": "Bearer ${GITHUB_TOKEN}" }
    },
    "local-db": {
      "type": "stdio",
      "command": "npx",
      "args": ["@bytebase/dbhub", "--dsn", "${DATABASE_URL}"]
    }
  }
}
```

**Scopes:**

- **Local** (default): stored in `~/.claude.json`, applies to current project only, personal.
- **Project**: `.mcp.json` at repo root, committed, applies to all team members.
- **User**: `~/.claude.json` user scope, applies to all your projects, personal.

**Authentication options:**

- Static: `headers: { "Authorization": "Bearer token" }`
- Dynamic: `headersHelper: "/path/to/script.sh"` (runs at connect time, stdout → headers)
- OAuth: automatic flow via `/mcp` command
- Environment expansion: `${VAR}` or `${VAR:-default}` in any field

**Nothing ships pre-configured.** You add all MCP servers yourself.

**MCP Tool Search (native, default on):**
Tool names load at startup; full schemas deferred until Claude needs to call them. Saves tokens on large MCP setups. Disable with `ENABLE_TOOL_SEARCH=false` or `alwaysLoad: true` per server.

**Security model:**

- Stdio servers run as user (no privilege escalation).
- Can be allow/denied via settings: `"Bash(mcp__github__*)"` in deny list.
- Prompt injection risk: if an MCP server fetches untrusted external content (e.g., a web-scraping tool), that content can inject instructions into Claude's context. Verify server trust before connecting.

### Genuinely Useful MCP Servers (Practitioner Reports)

- **GitHub MCP**: `mcp__github__*` — issues, PRs, comments, file contents, reviews. Replaces 80% of interactive `gh` CLI use. Boris's team uses this.
- **Sentry**: Error tracking and incident context. Boris's team.
- **BigQuery**: SQL operations without leaving Claude's context. Boris's team.
- **Slack**: Team communication integration. Boris's team.
- **Database tools** (`@bytebase/dbhub`): Direct DB access for schema exploration and queries.
- **Playwright/browser**: Automated UI interaction — core to the verify-app pattern.
- **Filesystem extended** (e.g., `mcp-server-filesystem`): When default file tools aren't enough.

### Global vs Project Scope

- **Global** (`~/.claude.json` user scope): Tools you use across every project — GitHub, Slack, generic database tools.
- **Project** (`.mcp.json`): Tools specific to a project's infrastructure — project-specific databases, internal APIs, deployment systems.

### Building Custom MCP Servers

Non-trivial. Uses the MCP protocol (modelcontextprotocol.io). Anthropic provides `mcp-server-dev` scaffold. SDK bindings for Node.js and Python. Complexity is comparable to building a small REST service.

Worth it when: Claude needs to interact with internal systems that have no public MCP server, or when you want type-safe structured tool calls rather than shell command wrapping.

## 5. Custom Commands and Subagents

### What's Real

**Skills / Commands:**

- `.claude/commands/name.md` → `/name` command (older format)
- `.claude/skills/name/SKILL.md` → `/name` command (preferred for new work, can bundle files)
- User-level: `~/.claude/commands/` or `~/.claude/skills/`

Frontmatter options:

- `description`: When to auto-invoke (Claude matches task description)
- `argument-hint`: Shows in `/help`
- `disable-model-invocation: true`: User-facing reference, no LLM invocation
- `user-invocable: false`: Claude-invoked only, not shown in `/help`

Parameterisation:

- `$ARGUMENTS`: Full argument string after the command name
- `$0`, `$1`, etc.: Positional argument access
- `!` prefix on a line: Execute as shell at invocation time, inject stdout as context

**Subagents:**

- `.claude/agents/name.md` or `~/.claude/agents/name.md`

Frontmatter:

```yaml
---
name: code-reviewer
description: Reviews code for correctness and security issues
tools: Read, Grep, Glob
model: claude-opus-4-8
memory: project
isolation: worktree
---
```

- `tools`: Restrict which tools this subagent can use (read-only reviewers, etc.)
- `model`: Override model for this subagent (run Haiku for fast tasks, Opus for deep review)
- `memory: project|user|local`: Persistent memory scope
- `isolation: worktree`: Auto-creates git worktree for full filesystem isolation

**Key distinction:**

- **Skills**: Prompt templates loaded into the current context window. Same session, same model, shared file access.
- **Subagents**: Independent execution contexts with separate context windows. Can have different tools, models, memory. Run in parallel. Isolated by default.

### Boris's Commands (community-reconstructed from interviews and bcherny-claude repo)

- `/commit-push-pr`: Git commit → push → PR in one command. Includes inline `!` bash to precompute git status (reduces model calls). Used dozens of times daily.
- `/quick-commit`: Fast commit without PR.
- `/test-and-fix`: Run tests, fix failures, loop.
- `/review-changes`: Pre-commit diff review.
- `/grill`: Adversarial code review — find problems.
- `/techdebt`: Cleanup sweep — identify technical debt without fixing it.

### Boris's Subagents (community-reconstructed)

- **code-simplifier**: Reviews completed code for quality, reuse, efficiency. Never changes *what* code does, only *how*. Used post-completion.
- **verify-app**: Detailed E2E testing instructions. Uses Claude Chrome extension to open browser, test UI, iterate until code works and UX feels right.
- **build-validator**: Ensures build passes all checks before a PR is created.
- **code-architect**: Evaluates architectural decisions.
- **staff-reviewer**: Adversarial architecture review.
- **oncall-guide**: Incident response workflows.

*Implementation specifics are inferred — the actual agent files are not publicly posted.*

### Best Practice

- Build `/commit-push-pr` or equivalent early — high ROI immediately.
- Build verify-app equivalent for your specific domain before diving into heavy implementation work.
- Use model override in subagent frontmatter: Haiku for fast/volume tasks, Opus for deep review.
- Tool restriction in subagent frontmatter prevents accidental side effects in review/analysis agents.
- Skills for workflows you run multiple times per day. Subagents for isolated, parallel, or restricted-tool operations.

## 6. Settings and Permissions

### What's Real

**Precedence (highest to lowest):**

1. Managed settings (MDM/OS/server policy — cannot be overridden)
2. CLI flags (`--permission-mode`, `--model`, `--effort`, etc.)
3. `.claude/settings.local.json` (personal overrides, auto-gitignored)
4. `.claude/settings.json` (project, committed)
5. `~/.claude/settings.json` (user-global)
6. Built-in defaults

Arrays (e.g., `permissions.allow`) **merge** across scopes. Scalars use most-specific value.

**Key fields:**

```json
{
  "permissions": {
    "allow": ["Bash(git *)", "Bash(npm test *)", "WebSearch", "mcp__github__*"],
    "deny":  ["Bash(rm -rf *)", "Bash(cat */.env*)"]
  },
  "hooks": {
    "PostToolUse": [{ "matcher": "Write|Edit", "hooks": [...] }]
  },
  "model": "claude-sonnet-4-6",
  "fallbackModel": ["claude-opus-4-8"],
  "effortLevel": "medium",
  "env": { "NODE_ENV": "development" },
  "autoMemoryEnabled": true,
  "autoMemoryDirectory": "~/.claude/projects",
  "claudeMdExcludes": ["**/legacy/**"],
  "editorMode": "vim",
  "minimumVersion": "1.5.0",
  "autoUpdatesChannel": "stable"
}
```

**When changes take effect:**

- **Immediately:** permissions, hooks, credentials, `apiKeyHelper`
- **Next session:** model, outputStyle, effortLevel

**Bash permission pattern syntax:**

- `Bash(git *:*)` — matches any git command (pattern is `command:args`)
- `Bash(npm test *)` — matches `npm test` with any args
- Non-Bash tools: just the tool name — `"WebSearch"`, `"mcp__github__*"`

### Best Practice

- **`~/.claude/settings.json`** (user-global): Safe read-only commands used everywhere — git read, ls, grep, find, cat (without secrets paths). Web search and standard MCP tools.
- **`.claude/settings.json`** (project, committed): Project-specific hooks, MCP servers, model defaults, allowed commands specific to this project's toolchain.
- **`.claude/settings.local.json`** (personal, gitignored): Personal preference overrides — editor mode, notification hooks, personal model preferences.
- Keep secrets out of settings files. Use environment variables or `apiKeyHelper` script.
- Allowlist specific commands rather than blanket `Bash(*)` — you want permission checks on destructive operations.

## 7. Cost and Token Management

### What Claude Code Does Automatically

**Prompt caching:**

- System prompt, CLAUDE.md, and repeated tool definitions are cached.
- Cache TTL: 5 minutes (subscriptions), expires with session (API key usage).
- Cache invalidated by: model switch, effort level change, fast mode toggle, MCP connect/disconnect, plugin enable/disable, permission deny rule changes.

**MCP Tool Search (default on):**

- Tool names load at startup. Full schemas deferred until Claude needs to call a tool.
- On large MCP setups (10+ servers), this saves significant tokens per session.

**Context compaction:**

- Triggered automatically when approaching context limit.
- Can also be triggered manually with `/compact`.
- CLAUDE.md re-injected from disk after compaction. Conversation state: summarised (lossy).

### RTK Already Installed

RTK handles 60–90% compression on Bash command output via PreToolUse hook. This is the single highest-impact token reduction available for CLI-heavy workflows. Already in place.

### Additional Strategies

- Use `/compact [hint]` **proactively at ~60% context**, not reactively at 90%. Provide steering hints to drop noise (verbose test output, debug logs) while keeping what matters.
- Delegate verbose operations to subagents: log analysis, full test runs, lengthy research. Subagent returns a summary; main context sees only the summary.
- Target CLAUDE.md under 200 lines.
- `/context` command shows current context composition — use it to identify what's consuming space.
- `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=70` triggers compaction earlier (practitioner experiment, not official).
- `/clear` between unrelated tasks rather than carrying stale context forward.

### Model Selection

| Model | Cost | Best For |
|---|---|---|
| Haiku 4.5 | ~$0.80/1M tokens | High-volume subagent tasks, fast summaries, simple code generation |
| Sonnet 4.6 | ~$3/1M | Default for most coding tasks — good balance |
| Opus 4.8 | ~$15/1M | Complex architecture decisions, multi-step planning, debugging subtle issues |

Set per-project defaults in `.claude/settings.json`. Use subagent `model:` frontmatter to run cheaper models for lightweight tasks within an expensive session.

Boris uses Opus for everything. This works if you're shipping 20–30 PRs/day and the throughput gain justifies the cost. For a solo dev at lower volume, Sonnet default with Opus for planning/architecture is the standard recommendation.

### Monitoring

- `/usage` — session cost and token breakdown.
- `stats-cache.json` in `~/.claude/` — aggregated historical costs.
- OTEL endpoint support for enterprise cost attribution.
- No built-in per-task cost tracking without external tooling.

## 8. GitHub and Git Integration

### What's Real

**Three integration paths:**

1. **GitHub MCP server** (local, interactive): Natural language interface to GitHub API within a Claude Code session. Issues, PRs, comments, file contents, reviews, labels. Uses `mcp__github__*` tools. No CLI syntax to memorise.

2. **GitHub Actions** (`anthropics/claude-code-action@v1`): Automated, server-side. Responds to `@claude` mentions in PR/issue comments. Posts reviews, creates PRs, runs code changes. Requires `ANTHROPIC_API_KEY` in repo secrets. Reads project `CLAUDE.md`. Supports Bedrock and Vertex AI for enterprise.

3. **`gh` CLI**: Shell-driven. Better for scripting and automation. More complete coverage of edge cases. Not the default in interactive sessions.

**MCP vs gh CLI:**

- MCP: use for interactive work within a session. Natural language, no context-switching.
- gh CLI: use for automation scripts, hooks, slash commands where structured shell output is needed.
- Boris's stated preference: `mcp__github__*` first; `gh` for what MCP can't do.

**GitHub Actions setup:**

```bash
/install-github-app  # sets up the GitHub App and workflow YAML
```

Or manually: install GitHub App, add workflow YAML, set `ANTHROPIC_API_KEY` secret.

Required repo permissions: Contents (R/W), Issues (R/W), Pull Requests (R/W).

**Automated PR review workflow:**

```yaml
name: Claude Review
on:
  pull_request:
    types: [opened, ready_for_review]
jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          prompt: "/code-review ${{ github.repository }}/pull/${{ github.event.pull_request.number }}"
```

Scope to `ready_for_review` (not `synchronize`) on busy repos to avoid burning API calls on every commit push.

### Best Practice

- Add GitHub MCP server globally so it's available in every project session.
- Use `@claude` GitHub Actions trigger for automated review on PRs — catches issues before human review.
- Slash command `/commit-push-pr` wraps the full workflow: run tests, commit, push, create PR. Pre-computes git status with `!` bash to reduce model calls.
- Pair worktrees with GitHub MCP: one worktree per feature, create PR directly from Claude within that session.

## 9. Testing and Verification

### What's Real

**Verify-app pattern (Boris Cherny):**

Core principle: *"Give Claude a way to verify its work. It 2-3x the quality of final results."*

Implementation: `.claude/agents/verify-app.md` subagent with detailed E2E test instructions. Uses the Claude Chrome extension to open a browser, test the UI, and iterate until code works and UX is right. For non-UI work: adapt to run test suite, probe API endpoints, run simulator.

The key: verification is built into the workflow definition, not into individual prompts. The agent knows *how* to verify; you just trigger it.

Boris: *"Never mark a task complete without proving it works. Ask: 'Would a staff engineer approve this?'"*

**Stop hook as quality gate:**

```json
{
  "Stop": [{
    "hooks": [{
      "type": "command",
      "command": "bash -c 'INPUT=$(cat); if [ \"$(echo \"$INPUT\" | jq -r .stop_hook_active)\" = \"true\" ]; then exit 0; fi; npm test --silent || exit 2'",
      "timeout": 60
    }]
  }]
}
```

Converts the quality check from advisory (CLAUDE.md instruction) to deterministic (hook). `stop_hook_active` guard prevents infinite loops.

**Karpathy's Goal-Driven Execution:**
Define success criteria before starting. Claude loops until verified. This is TDD applied at the prompt level: write acceptance criteria first, let Claude write code to satisfy them. Strong criteria → independent iteration. Weak criteria ("make it work") → ongoing clarification cycles.

### Best Practice

1. Define acceptance criteria before handing off any non-trivial task.
2. Build a verify-<domain> subagent that knows how to test your specific project. Keep it updated as the project evolves.
3. Stop hook with test suite check — invest in this once you have a stable test harness.
4. Build-validator subagent for pre-PR validation.
5. Don't just check that tests pass. The verify-app model checks that the *thing actually works for a user* — functionally different.

## Highest-Impact Changes — Priority Summary

Ordered by impact. A solo dev starting from a reasonable baseline (which Clarke has).

**Immediate (do before any major implementation work):**

1. **Add build/test/lint/run commands to CLAUDE.md.** Highest ROI of any CLAUDE.md content. Claude can't guess these. Every session where they're missing wastes turns on wrong commands.

2. **Build a verify-<domain> subagent.** Boris calls it the single highest-leverage thing you can do. 2–3x quality improvement. The verify-app pattern scales beyond UI testing — define it for your domain (API probing, test suite, simulation, etc.).

3. **Errors → CLAUDE.md updates, always.** When Claude makes a mistake, don't correct it in conversation — add a rule to CLAUDE.md. Compounding benefit across sessions. Boris does this *every single time*.

**High leverage (first week):**

1. **Convert the most-violated CLAUDE.md rules to hooks.** Identify rules that keep getting ignored and move them to PreToolUse/PostToolUse. 100% deterministic vs ~80% adherence.

2. **Stop hook that runs tests before session ends.** Once test suite exists. Prevents shipping broken code silently.

3. **Set up git worktrees for parallel tasks.** `claude --worktree name` is low-friction. Enables working multiple tasks in parallel with full isolation.

4. **Desktop notifications hook.** Required for parallel session management. Without it, parallel sessions stall waiting for input you don't notice.

**Team-level (before adding collaborators):**

1. **Commit hooks and MCP config to `.claude/settings.json`.** Every team member gets the same workflow. Team CLAUDE.md review in PRs.

2. **GitHub MCP server globally.** Replaces most interactive `gh` CLI use. Enables natural language GitHub operations within sessions.

3. **Handoff brief discipline.** Write a brief before closing long sessions. Under 500 tokens. Saves hours of re-establishing context. Consider a `/handoff` slash command that generates it.

**Ongoing:**

1. **Compact at 60% context, not 90%.** Quality degrades well before the hard limit. Provide steering hints to preserve what matters.

2. **Per-task model selection in subagent frontmatter.** Haiku for fast/high-volume tasks, Sonnet for default, Opus for architecture and deep debugging.

3. **`@reference` pattern for large specs.** Keep CLAUDE.md under 200 lines by externalising extended specs. Load on demand rather than loading everything on startup.
